## 🟢 第一部分：项目核心背景概念普及 (用于铺垫)

### 1. 什么是局部特征匹配 (Local Feature Matching)？
给定同一场景但不同视角（例如你站在桌子左边和右边拍的同一张桌子）的两张照片，计算机需要自动找出这两张照片中**物理上对应的同一个像素点**（例如桌角）。这在计算机视觉中被称为局部特征匹配，它是 3D 重建、SLAM（即时定位与建图）、以及自动驾驶视觉定位的基石。

### 2. 什么是 DINOv3 与 Zero-Shot？
**DINOv3** 是基于 Vision Transformer (ViT) 架构的自监督视觉基础模型。它在海量图像上进行了无监督学习，提取的特征不仅包含像素级别的颜色信息，更包含极强的**高层语义信息**（它不仅知道那是一个角，还知道那是“木头桌子”的角）。
**Zero-Shot（零样本）** 能力是指：我们不需要针对“特征匹配”这个任务对 DINOv3 进行任何重新训练，直接提取它的 1024 维输出特征计算相似度，就能实现非常不错的匹配精度。本项目中，Zero-Shot 是我们的“保底分数线”。

### 3. 什么是对比学习 (Contrastive Learning) 与 InfoNCE 损失？
我们希望对 DINOv3 进行**有监督微调 (Fine-Tuning)**，让它不仅懂语义，更懂“对极几何”（几何结构的匹配）。我们采用的损失函数是 InfoNCE：
- **正样本 (Positive)**：同一物理点的两个视角的特征，我们要在数学空间上把它们**拉近**。
- **负样本 (Negative)**：不同的物理点，我们要把它们在空间中**推远**（使其互相正交）。

## 🟢 第二部分：Slide by Slide 详细讲稿与演进历程

### Slide 3：项目第一阶段 (Phase 1) —— Zero-Shot 基线测试
在项目的第一阶段，由我们组的 **lmz** 同学负责牵头，测试了原生 DINOv3 模型在完全不经过微调的情况下的“裸跑”能力（即 Zero-Shot）。
- **工作内容**：编写了底层的特征提取与 MNN 互相最近邻匹配脚本，并接入了评估管道。
- **评测指标**：我们重点关注了 Matching Precision (匹配中满足对极几何约束的内点比例) 和 Pose AUC (基于估计出的本质矩阵计算出的角度误差)。
- **结果与意义**：即使没有任何针对性的训练，DINOv3 在 NAVI 上依然跑出了约 49% 的 Precision，在 ScanNet 上达到了 32%。这从根本上证明了视觉大模型的强大泛化能力。这一阶段的工作为整个项目划定了非常坚实的**基线分数**。如果我们的微调不能超过这个分数，那就说明微调方向错了。

### Slide 4：项目第二阶段 (Phase 2) —— 开启微调与底层的填坑
接下来进入微调阶段。但在正式微调前，我们在接入原有代码管道时，排查并修复了严重的工程级 Bug：
1. **坐标系缩放错位**：评估代码用的尺寸是 `640x480`，但训练/提取时的图像池化尺寸是 `448x448`。由于原代码缺失了映射逻辑，导致送入计算的匹配坐标全盘错位。我们重写了数学映射关系。
2. **训练管道卡死**：原代码在读取超高分辨率图片时因为单线程加载导致 I/O 阻塞死锁。我们优化了 DataLoader 和缓冲策略，并在 NAVI 数据集上清洗并生成了 5500+ 对干净的训练样本，彻底清除了工程障碍。

### Slide 5：第一次微调滑铁卢 —— 最后两层微调 + Projection Head
由于 ViT-Large 模型极其庞大（超 300M 参数），在普通的 8GB 显卡上进行全参微调会立刻导致 OOM（内存溢出）。
**我们的初步方案**：冻结前 22 层，开放最后 2 层，并在末尾追加一个随机初始化的两层全连接网络（Projection Head），试图将原生 1024 维特征压缩到 256 维。
**结果是灾难性的**：微调后，两个数据集上的匹配精度从 Zero-Shot 的 ~40% 直接**暴跌至 3% 左右**。
**深度诊断**：在非常有限的硬件算力下，这个随机初始化的 Projection Head 根本没有时间收敛，它不仅没有学到几何特征投影，反而像一个“瞎子”一样，把 DINO 原本包含的优秀 1024 维语义信息彻底破坏成了随机噪声。

### Slide 6：架构全面升级 —— 引入 LoRA (Low-Rank Adaptation)
痛定思痛，我们决定引入大语言模型微调中最前沿的 **LoRA** 技术：
1. 我们**砍掉了破坏性的 Projection Head**，让模型直接输出原生的 1024 维特征。
2. 我们在全部 24 层注意力机制的 QKV 矩阵旁，并联了小参数量的 LoRA 适配器。
3. **最巧妙的设计 (B=0 初始化)**：我们在代码中严格约束了初始化状态。这意味着在训练的第 0 步，LoRA 旁路的输出为 0。这就保证了：**模型在微调开始的那一刻，其输出 100% 等同于 Zero-Shot 的强力表现**。我们期望在此基础上，只做极其微小的几何增量学习，稳赚不赔。

**【工作量展示 / 核心代码讲解】**
*建议在 PPT 中展示我们自己编写的 LoRA 核心类，重点突出 `torch.zeros` 的初始化设计，向老师证明我们深入了底层模型架构。*
```python
class LoRALinear(nn.Module):
    def __init__(self, original: nn.Linear, rank: int = 4, alpha: float = 1.0):
        super().__init__()
        # 冻结原始的 DINOv3 权重
        self.original = original
        self.original.weight.requires_grad = False
        
        # 核心：B 矩阵全零初始化！保证初始状态无破坏
        self.lora_A = nn.Parameter(torch.empty(rank, original.in_features))
        self.lora_B = nn.Parameter(torch.zeros(original.out_features, rank))
        nn.init.kaiming_uniform_(self.lora_A)

    def forward(self, x: torch.Tensor) -> torch.Tensor:
        # 最终输出 = 冻结的原始输出 + LoRA 微调增量
        out = self.original(x)
        lora_out = (x @ self.lora_A.t()) @ self.lora_B.t() * self.scaling
        return out + lora_out
```

### Slide 7-8：惊天发现 —— 诡异的二次坍塌与损失函数悖论 (核心高光)
满怀信心地开启 LoRA 微调后，结果竟然是：**模型依然全面崩溃（精度约 5%）**！既然初始状态已经和 Zero-Shot 一模一样，且没有随机层破坏特征，为什么还会坍塌？

**【可视化定性证明：PCA 特征降维图】**
*为了让观众更直观地理解坍塌，我们编写了 PCA 可视化脚本，将 1024 维特征降维到 RGB 3 维并映射到原图上（参考 `presentation/result/pca_zero_shot.png` 和 `pca_lora_collapsed.png`）。*
- *在 Zero-Shot 阶段，PCA 图展现了极其丰富的渐变色彩，不同的语义结构（比如桌子边缘、物体轮廓）分界非常明显。*
- *而在 LoRA 坍塌阶段，PCA 图彻底变成了一片均一的色块。这意味着模型所有的输出特征都变成了完全相同的死板常量。*

我们深入挖掘了训练日志，发现了一个诡异的现象：**训练 Loss 死死卡在 `4.855` 无法下降**。
通过严格的数学反推：
- 我们采样的最难负样本数 `K = 128`。
- 若模型发生彻底摆烂（即不管输入什么，模型都输出一模一样的常数），InfoNCE Loss 的交叉熵计算将退化为均匀分布的香农熵：`Loss = ln(负样本数 + 正样本数)`。
- **数学等式**：`ln(128 + 1) = ln(129) ≈ 4.8598`！
**这个数字的完美吻合，在数学上铁证如山地表明：模型发生了 Mode Collapse（模式坍塌）。问题根本不在我们设计的网络架构，而在底层的损失函数！**

**为什么会坍塌？（损失函数悖论）**
受限于本地机器 RTX 5060 (8GB) 的显存，我们被迫设置 `batch_size = 1`。这意味着，对比学习所需的负样本，只能从**同一张图片**的其他位置去取（单图局部对比，Intra-Image InfoNCE）。
这产生了一个荒谬的理论悖论：
- DINOv3 知道“同一面白墙上的相邻像素，它们的特征应该长得一样”（语义连续性）。
- 但我们的 Loss 函数却强迫：“同一面白墙上的相邻 256 个点互为负样本，你们的特征必须互相正交、推得越远越好”。
**面对这种“不可理喻”的矛盾惩罚，神经网络选择了最聪明的逃避方式：直接罢工，把所有特征输出为相同的常量，吃下 `ln(129)` 的保底惩罚**。这就是两次坍塌真正的幕后黑手！

### Slide 9：最终解法：算法突围硬件瓶颈 (Safe Radius)
如果在顶级实验室（拥有 24G 显存），我们只需把 `batch_size` 开到 16，让负样本来自**不同的图片（Inter-Image）**，这个悖论就会自然解开（不同的图片之间本就应该互斥）。

但在算力受限（8GB显存）的情况下，我们决定用算法打败硬件！
我们深入底层重写了 `finetune/loss.py` 中的损失函数计算图，独创了 **Safe Radius (安全面罩)** 算法：
1. 每次采样前，动态计算图片内所有特征点的欧式物理距离。
2. 如果两个点物理距离 `< 80 像素 (5 个 Patch)`，我们就构建一个无穷大的掩码（Mask），将它们从负样本堆中**强行剔除**。
**逻辑非常简单但极其有效**：项目组不再强迫模型把物理上相邻（本就相似）的特征推开。

**【工作量展示 / 核心代码讲解】**
*建议在 PPT 中展示我们重写的 Loss 函数中的掩码（Masking）机制，这是打破模型坍塌的终极钥匙，极具学术价值。*
```python
# Safe Radius Masking (打破单图 InfoNCE 悖论的核心算法)
if idx_a is not None and self.safe_radius > 0:
    # 1. 动态计算二维空间坐标 (x, y)
    x_a = (idx_a % w_patches).float()
    y_a = (idx_a // w_patches).float()
    
    # 2. 计算特征点对之间的欧几里得距离平方
    dist_sq = (x_a.unsqueeze(1) - x_a.unsqueeze(0))**2 + \
              (y_a.unsqueeze(1) - y_a.unsqueeze(0))**2
              
    # 3. 核心：如果距离小于安全半径，构建掩码
    mask = (dist_sq < self.safe_radius**2)
    mask.fill_diagonal_(False) # 排除对角线（正样本自己）
    
    # 4. 强制替换为 -inf，使其永远不会被选为最难负样本
    sim.masked_fill_(mask, float('-inf'))
```

**进展成果**：修改后进行测试，Loss 终于成功跌破了 `4.855` 的死亡线，开始平稳下降。我们用纯算法手段，在苛刻的硬件约束下，成功拆除了对比学习在密集特征匹配任务上的理论级地雷。
