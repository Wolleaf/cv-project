## 实验总览

本项目从 2026 年 4 月开始，历经六个阶段、五种微调方案、两个数据集的系统实验。核心目标是回答一个看似简单的问题：

> 用对比学习微调 DINOv3，能否提升其在局部特征匹配任务上的位姿估计精度？

答案是：**不能。任何形式的对比微调都只会让精度下降。**

以下按时间顺序记录完整的实验历程——包括成功的诊断和失败的实验，因为在这个项目中，"为什么失败"比"如何成功"更有价值。

## 阶段 0：Zero-Shot 基线

### 0.1 目标

搭建完整的 DINOv3 + MNN + Essential Matrix 评估 pipeline，测量不经过任何微调的 Zero-Shot 特征匹配精度。这个基线是所有后续微调实验的对照基准。

### 0.2 Pipeline

```
图像对 (A, B)
    ↓ resize → 448×448
    ↓ DINOv3 ViT-L/16 提取密集特征
    ↓ 28×28 = 784 个 patch，每个 1024-dim
    ↓ L2 归一化
    ↓ MNN 匹配（双向最近邻）
    ↓ 输出匹配点对 → CSV
    ↓ OpenCV USAC 估计 Essential Matrix
    ↓ SVD 分解位姿 → 计算角度误差
    ↓ 统计 AUC@5/10/20 + Precision
```

### 0.3 数据集

**NAVI**（物体级）：
- 3000 对测试图像
- 单个物体的多视角照片（转盘式拍摄）
- 图像分辨率：最长边 1024 像素
- 测试文件：`datasets/navi_with_gt.txt`

**ScanNet**（场景级）：
- 1500 对测试图像
- 室内场景扫描（办公室、客厅、走廊等）
- 图像分辨率：640×480
- 测试文件：`datasets/scannet_with_gt.txt`

### 0.4 基线结果

| 数据集 | AUC@5 | AUC@10 | AUC@20 | Precision |
|--------|-------|--------|--------|-----------|
| NAVI | 0.24 | 1.07 | 3.31 | 49.17% |
| ScanNet | 0.23 | 1.18 | 4.44 | 32.20% |

### 0.5 基线分析

- NAVI 的 Precision（49.17%）远高于 ScanNet（32.20%）——物体级场景的纹理更丰富，语义更明确
- 两者的 AUC@5 都极低（< 0.25%），说明极精确的位姿估计（< 5° 误差）对 DINO 特征来说非常困难
- AUC@20 分别为 3.31% 和 4.44%——即使是宽松的 20° 阈值，也只有很少的图像对能恢复出正确的位姿
- Precision 与 AUC 的关系并不简单：ScanNet 的 AUC@20 更高，但 Precision 更低

### 0.6 初始假设（后被证伪）

> DINOv3 的特征已经展现出匹配能力（Precision 32-49%）。如果能用几何对应关系作为监督信号进行对比学习微调，让"对应 patch"的特征更相似、"非对应 patch"的特征更不相似，匹配精度应该会进一步提升。

这个假设看起来非常合理——实际上它被几乎所有对比学习文献的逻辑所支持。但它基于一个隐含的前提：**匹配任务需要的就是"对应 patch 相似、非对应 patch 不相似"**。接下来的六个阶段将证明这个前提在密集匹配中是错误的。

## 阶段 1：Projection Head + InfoNCE（第一轮失败）

### 1.1 实验设计

**日期**：2026 年 4 月（第一周）

**假设**：冻结 DINO backbone 的大部分层（前 22 层），在顶部添加可训练的投影头（1024 → 640 → 256），用 InfoNCE 损失拉近对应 patch 的特征。

**架构**：
```
DINOv3 backbone (层 1-22 冻结, 层 23-24 可训练)
    ↓
Projection Head:
    Linear(1024 → 640) → BatchNorm → ReLU
    Linear(640 → 256)
    ↓
256-dim 特征 → L2 归一化 → InfoNCE 损失
```

**训练配置**：
| 参数 | 值 |
|------|-----|
| Batch size | 1（单对图像） |
| 负样本数 | 128（从同一图像对采样） |
| 温度 τ | 0.07 |
| 学习率 | 1e-3 |
| Epochs | 15 |
| GPU | RTX 5060 8GB |
| 训练数据 | NAVI 5596 对, ScanNet 1500 对 |

### 1.2 训练过程

训练顺利进行——损失函数稳定下降。第 1 个 epoch 结束时的 loss 约为 4.86，之后基本不再变化。

### 1.3 结果

| 数据集 | AUC@5 | AUC@10 | AUC@20 | Precision |
|--------|-------|--------|--------|-----------|
| NAVI | 0.00 | 0.03 | 0.10 | 3.16% |
| ScanNet | 0.00 | 0.07 | 0.25 | 3.47% |

**Precision 从 49.17% 暴跌至 3.16%（NAVI），下降了 93.6%！**

### 1.4 诊断：Mode Collapse 的数学证明

注意到训练 loss 精确地稳定在 4.86 附近。计算 InfoNCE 在随机猜测情况下的期望值：

```
当所有特征退化为均匀随机向量时：
  - 正样本的相似度和 K 个负样本的相似度没有区分
  - 每个被选中的概率都是 1/(K+1)
  - InfoNCE → -log(1/(K+1)) = ln(K+1)

当 K = 128 时：
  L = ln(129) ≈ 4.859812...
```

**我们的训练 loss 精确地收敛到了这个值。** 这意味着模型找到了一个"万无一失"的策略来最小化 loss：把所有特征都变成随机噪声。当正样本和所有负样本无法区分时，loss = ln(K+1)——不是最优但稳定，没有梯度可以进一步降低。

这是经典的 **Mode Collapse（模态坍缩）**——对比学习中最著名的失败模式。

### 1.5 初步理解（后来证明不够深入）

当时我们认为失败的原因是：
1. 投影头的随机初始化破坏了 DINO 的预训练特征
2. batch_size=1 导致所有负样本来自同一图像，存在大量虚假负样本（语义相同但位置不同的 patch 被当作负样本）

基于这个理解，我们转向了 LoRA。

## 阶段 2：LoRA + InfoNCE（第二轮失败）

### 2.1 实验设计

**日期**：2026 年 4 月（第二周）

**关键改进**：
1. 放弃投影头，特征保持在原始 1024-dim DINO 空间
2. 使用 LoRA 替代全层微调，B 矩阵零初始化保证训练起点 = Zero-Shot

**架构**：
```
DINOv3 backbone (所有层冻结)
    ↓ + LoRA (注入所有 QKV + Proj 层)
    ↓ 24 层 × 4 矩阵 × (1024×4 + 4×1024) = ~0.39M 可训练参数
    ↓ 1024-dim 特征（L2 归一化）
    ↓ InfoNCE 损失
```

**为什么 LoRA 应该有用**：
- B=0 初始化 → 训练开始时模型输出**完全等于** Zero-Shot DINOv3
- 理论上最差情况是不学习 → 退回 Zero-Shot → Precision 应该 ≥ Zero-Shot
- ~0.39M 参数 → 轻量，难以严重过拟合

### 2.2 训练配置

| 参数 | 值 |
|------|-----|
| Batch size | 1 |
| 负样本数 | 128 |
| LoRA rank | 4 |
| 学习率 | 1e-3 |
| Epochs | 15 |
| GPU | RTX 5060 8GB |

### 2.3 结果

**ScanNet**（当时的实验重点）：

| 指标 | Zero-Shot | LoRA + InfoNCE |
|------|-----------|----------------|
| AUC@5 | 0.23 | 0.00 |
| AUC@10 | 1.18 | 0.09 |
| AUC@20 | 4.44 | 0.96 |
| Precision | 32.20% | 5.51% |

**再次坍塌！即使起点完全等于 Zero-Shot，模型仍然在学习过程中主动退化。**

### 2.4 深度诊断：为什么 LoRA 也没救

这迫使我们重新审视 InfoNCE 在密集匹配中的根本问题。通过仔细分析负样本的组成，我们发现：

**在同一张图像中（batch_size=1 的情况）：**
- 一面白墙可能覆盖 50+ 个 patch
- 这 50 个 patch 的 DINO 特征天然高度相似（语义上它们都是"白墙"）
- InfoNCE 随机选 128 个负样本，其中大量来自同图像的相似区域
- 模型被命令："让正样本比这些'白墙'负样本更相似"
- 但正样本本身可能就是"白墙"的一部分 → 与负样本看起来几乎一样
- 模型唯一的出路：**破坏语义结构，让所有特征变得随机** → 这样就没有 patch 天然相似了

**关键认识**：问题不在于"负样本包含了一些错误的"，而在于**负样本对于一个在连续语义流形上的任务来说，本身就是个错误的概念**。

## 阶段 3：LoRA + Safe Radius（最接近成功的一次）

### 3.1 实验设计

**日期**：2026 年 4 月（第二周）

**假设**：InfoNCE 对密集匹配的主要破坏来自**空间相邻的负样本被强行推开**。如果在空间上保护相邻 patch，模型就能在保持局部语义连续性的同时学习跨图像对应。

### 3.2 Safe Radius 算法

```
对每个正样本 patch（锚点）：
    对每个候选负样本：
        计算空间距离 = sqrt((row_a - row_n)² + (col_a - col_n)²)
        如果 距离 < safe_radius (5 patches)：
            将该负样本从 InfoNCE 的 softmax 分母中排除（mask = -inf）
```

Safe Radius = 5 patches = 80 像素（在 448×448 图上）= 约图像宽度的 18%。

**为什么是 5**：过小保护不足（1-2 patches 无法覆盖纹理区域），过大保护过度（可能误排除真正的负样本）。5 是一个在经验上合理的值。

### 3.3 训练配置

| 参数 | 值 |
|------|-----|
| Batch size | 1 |
| Safe Radius | 5 patches |
| 其他 | 与阶段 2 相同 |

### 3.4 结果

**ScanNet**（本阶段重点验证）：

| 指标 | Zero-Shot | LoRA (无 Safe Radius) | LoRA + Safe Radius |
|------|-----------|----------------------|-------------------|
| AUC@5 | 0.23 | 0.00 | 0.00 |
| AUC@10 | 1.18 | 0.09 | 0.26 |
| AUC@20 | 4.44 | 0.96 | 1.84 |
| Precision | 32.20% | 5.51% | **28.83%** |

### 3.5 分析

**正面**：
- Precision 从 5.51% 大幅恢复到 28.83%，仅比 Zero-Shot 的 32.20% 低 10.5%
- 证明了"不合理的负样本"确实是退化的重要原因
- Safe Radius 有效地保护了局部语义连续性

**负面**：
- AUC 仍然远低于 Zero-Shot（AUC@20: 1.84 vs 4.44）
- **Precision 恢复了，AUC 没有**——这是一个极其重要的信号

### 3.6 Precision 与 AUC 的分化——关键诊断信号

Precision 高但 AUC 低意味着：
- 单个匹配质量尚可（对极几何约束被满足）
- 但匹配点的**空间分布**很差
- 位姿估计需要匹配点在图像上均匀分布——如果所有匹配都集中在图像中心的一小块区域，位姿估计会严重退化

**这个分化是对比学习破坏特征空间结构的最直接证据：**
- Safe Radius 保护了局部语义连续性 → 局部匹配质量恢复（Precision ↑）
- 但 InfoNCE 仍在推开语义相似但空间不相邻的 patch（不同位置的白墙） → 全局语义结构仍被破坏
- 语义结构被破坏 → 匹配点集中在少数"幸存"的语义区域 → 空间分布差 → 位姿估计退化（AUC ↓）

## 阶段 4：RTX 5090 大 Batch 验证

### 4.1 实验设计

**日期**：2026 年 4 月（第三周）

**假设**：batch_size=1 迫使所有负样本来自同一图像。增大 batch_size 引入跨图像负样本（Inter-Image Negatives），不同图像的白墙不会构成虚假竞争。

### 4.2 环境迁移

| 资源 | 之前（本地） | 现在（云端） |
|------|------------|------------|
| GPU | RTX 5060 | RTX 5090 |
| 显存 | 8 GB | 32 GB |
| Batch size | 1 | 8 |
| 负样本来源 | 仅同图 | 同图 + 跨图 |

### 4.3 结果

用户的反馈直接而明确：

> "结果还是不太行。"

##

### 4.4 分析

排除显存瓶颈后，问题仍然存在。这意味着 InfoNCE 的软肋不仅仅是"同图负样本"，而是更深层的东西。

**InfoNCE 在密集匹配中的三个不可修复的缺陷**：
1. τ=0.07 将相似度从 [0,1] 放大到 [0,14.3]，softmax 退化为近似 one-hot
2. 密集匹配中，一个正样本面对数百个语义相似的负样本——softmax 强制只选一个赢家
3. "所有负样本被推开"这个操作本身与语义连续流形相矛盾

增大 batch_size 能缓解问题 2（更多样化的负样本），但不能解决 1 和 3。

## 阶段 5：StableMatchingLoss（终极验证）

### 5.1 实验设计

**日期**：2026 年 4 月（第三周）

**假设**：如果 InfoNCE 的问题是 softmax 的硬竞争，那就彻底放弃 softmax。设计一个全新的 margin-based 损失函数。

### 5.2 StableMatchingLoss 设计

```
L_total = L_pos + λ_neg·L_neg + λ_div·L_div + λ_preserve·L_preserve

组件：
  L_pos      = (1 - s_pos)²                    # 温和吸引正样本
  L_neg      = ReLU(margin - s_pos + s_hn)    # 只推开最难负样本
  L_div      = mean(off-diag(sim_matrix))      # 防止同一图像的特征坍缩
  L_preserve = ||f_finetune - f_pretrained||²  # 防止偏离预训练太远

超参数：
  margin = 0.3
  λ_neg = 1.0
  λ_div = 0.1
  λ_preserve = 0.5
```

### 5.3 关键设计理由

**L_pos 使用平方损失而非 hinge**：hinge loss (max(0, m - s_pos)) 要求 s_pos > m，这意味着正样本对必须有高相似度。但如第 5 节所述，正样本对（不同视角的同一 3D 点）本就不应该有完全相同的特征。平方损失只是温和地鼓励相似，不强求。

**L_neg 只取最难负样本**：密集匹配中有几百个负样本，绝大多数天然与正样本不相似。推开它们没有信息量，只会破坏语义结构。只关注那个"伪装者"——与正样本相似但不对应的最难区分者。

**L_preserve 防止灾难性遗忘**：保持微调后的特征接近预训练的 DINO 特征，防止在数千张微调图片上丢失 1.42 亿张图片学到的知识。

### 5.4 训练配置

| 参数 | 值 |
|------|-----|
| Batch size | 8 |
| Epochs | 15 |
| LoRA rank | 4 |
| 学习率 | 5e-4 |
| GPU | RTX 5090 32GB |

### 5.5 结果

| 数据集 | AUC@5 | AUC@10 | AUC@20 | Precision |
|--------|-------|--------|--------|-----------|
| NAVI | 0.00 | 0.02 | 0.11 | 5.13% |
| ScanNet | 0.04 | 0.26 | 1.43 | 10.97% |

### 5.6 分析：最后一根稻草

**StableMatchingLoss 不仅没有超越 Zero-Shot，甚至不如 LoRA+Safe Radius（28.83%）——差了近 3 倍。**

这彻底排除了"损失函数形式"这个变量。

| 变量 | 是否被排除 | 证据 |
|------|-----------|------|
| 损失函数形式 | ✅ 已排除 | InfoNCE 和 StableMatchingLoss 都失败 |
| 温度参数 | ✅ 已排除 | StableMatchingLoss 没有温度参数 |
| Softmax 竞争 | ✅ 已排除 | StableMatchingLoss 不使用 softmax |
| Batch size | ✅ 已排除 | bs=8 在 RTX 5090 上充分验证 |
| 显存 | ✅ 已排除 | 32GB 绰绰有余 |
| 训练数据量 | ✅ 已排除 | NAVI 5596 对仍然失败 |
| 初始化 | ✅ 已排除 | LoRA B=0 保证起点 = Zero-Shot |
| Pipeline bug | ✅ 已排除 | Zero-Shot 结果稳定且合理 |
| 学习率 | ✅ 已排除 | 1e-3 到 5e-4 都试过 |

**结论：问题不在工程变量。必定在方法论层面。**

### 5.7 核心洞察的确立

经过五个阶段的系统验证，我们确立了以下不可动摇的事实：

> 任何形式的对比微调（将对应 patch 拉近、非对应 patch 推开），无论使用何种损失函数、何种架构、何种 batch size，都在系统性地破坏 DINOv3 的匹配能力。

原因不是工程实现的问题，而是：
1. DINOv3 的特征是**语义的**——"门把手 ≈ 门把手"
2. 匹配需要的是**几何的**——"这个门把手的左上角 ≠ 那个门把手的左上角"
3. 对比学习的"拉近对应、推开非对应"操作在语义空间上制造裂缝
4. 裂缝破坏了 Zero-Shot 匹配赖以工作的语义平滑性
5. 少量的微调数据（< 0.001% 的预训练数据）无力修复这些裂缝

## 阶段 6：Matchability Predictor — 路线 A

### 6.1 思路转折

**日期**：2026 年 4 月（第四周）

基于前五个阶段的教训，我们换了一个完全不同的思路：

> **DINOv3 特征完全不动。只训练一个极小的网络来预测每个 patch 是否"值得匹配"。**

```
之前的思路：修改特征 → 让"好的"更相似、"坏的"更不相似 → [失败]
路线 A：    不改特征 → 判断每个 patch 是否可匹配 → 只在高分 patch 上匹配
```

### 6.2 理论基础

DINOv3 的匹配错误主要来自纹理缺失和重复纹理区域。如果能识别并过滤这些区域的 patch，正确匹配占比自然提升。

这是**零风险**的思路：backbone 完全冻结，Predictor 退化为输出均匀分数时 → 完全等于 Zero-Shot。不存在负优化空间。

### 6.3 模型设计

```
DINOv3 1024-dim 特征 (冻结, 不参与训练)
    ↓
Linear(1024 → 256) → BatchNorm → ReLU
    ↓
Linear(256 → 64) → BatchNorm → ReLU
    ↓
Linear(64 → 1) → Sigmoid
    ↓
可匹配度分数 ∈ [0, 1]
```

- 参数量：~280K
- 输入：冻结的 DINO 特征（不需要图像，只需要特征向量）
- 训练速度：< 10 分钟/epoch（RTX 5090）

### 6.4 训练标签生成

不使用外部监督（不需要深度图、不需要位姿真值）。直接用冻结 DINO 的匹配结果自生成标签：

```
对每对训练图像：
    1. DINO 提取特征（冻结）
    2. MNN 匹配 + 双向互检
    3. 余弦相似度 > 0.5 且通过互检 → 标签 1（可匹配）
    4. 匹配失败 或 不满足互检 → 标签 0（不可匹配）
```

类平衡采样：训练时正负样本按 2:1 比例采样，防止类不平衡。

### 6.5 推理流程

```
图像 A → DINO(冻结) → 784 个特征 → Predictor → 784 个分数
    ↓
按分数排序，保留 top K 个 patch
    ↓ K = 784 × keep_ratio
    ↓ 例如 keep_ratio=0.5 → 保留 392 个高分 patch
    ↓
只在高分 patch 上做 MNN 匹配
    ↓
输出 CSV → 评估
```

### 6.6 测试计划

测试 5 个 keep_ratio（0.8, 0.6, 0.5, 0.4, 0.3），找出最优过滤比例。预期：
- 过滤过少（ratio=0.8）：保留太多不可靠 patch → 效果提升有限
- 过滤适中（ratio=0.4-0.6）：过滤掉纹理缺失/重复区域 → Precision 显著提升
- 过滤过多（ratio=0.3）：丢失太多匹配点 → 位姿估计不稳定

### 6.9 Route A 失败分析

Precision 从 Zero-Shot 的 49%/32% 暴跌至 5-6%。Predictor 不仅没有过滤掉坏 patch，反而把好 patch 过滤掉了。

**根因**：Predictor 只能看到单个 patch 的 1024 维特征。但一个 patch 能否成功匹配，取决于它在整个图像的特征分布中是否**独特**——这个信息不存在于单个 1024 维向量中。一个白墙 patch 和另一个白墙 patch 的特征几乎相同，Predictor 无法区分它们。

> **关键教训**：对于匹配任务，"可匹配性"是**上下文属性**而非**局部属性**。只有 Self-Attention 能让 patch 感知到自己在整体分布中的位置。

## 阶段 7：Route B — Frozen DINO + 注意力匹配头（LAST ATTEMPT）

### 7.1 思路

**日期**：2026 年 4 月（第四周）

Route A 的失败揭示了一个关键事实：**单个 patch 无法判断自身的匹配价值**。解决方案是 Self-Attention——让每个 patch 在更新特征之前看到同一图像的所有其他 patch。

### 7.2 架构

```
DINOv3 (冻结, 1024-dim)
    ↓
Linear Projection: 1024 → 256 (共享权重)
    ↓
+ 2D Sinusoidal Positional Encoding
    ↓
4× 迭代 [Self-Attn(A) → Self-Attn(B) → Cross-Attn(A↔B) → FFN]
    ↓
Final Linear → L2 Normalisation
    ↓
Dual-Softmax Matching → 输出匹配
```

**参数量**：~1M，d_model=256, nhead=4, 4 blocks

### 7.3 训练设计

- **损失**：Dual-Softmax NLL — 直接在匹配分配矩阵上优化，而非特征相似度
- **优化器**：AdamW + OneCycleLR
- **训练数据**：与之前相同（NAVI 5596 对 with depth，ScanNet 1500 对 with epipolar）
- **训练配置**：batch_pairs=4, epochs=20, lr=1e-3, wd=1e-3

### 7.4 为什么这次不一样

与 Route A 的根本区别：
- Route A: `f(desc_i)` — 单 patch 判断
- Route B: `f(desc_i, all_descs_A, all_descs_B)` — 全图上下文判断

与对比微调的根本区别：
- 对比微调：修改特征空间 → 破坏语义结构
- Route B：不改特征空间 → 只学习"如何匹配"→ 语义结构完整保留

```bash
./run_matching_head_5090.sh                # 完整流程
./run_matching_head_5090.sh --skip-train   # 跳过训练
```

## 根因分析总结

本项目的根因分析已在 `learn.md` 第 5-6 节中详细展开，此处做最简总结。

### 四层根因（由浅入深）

**第一层：训练标签与 DINO 特征的粒度不匹配**
16×16 的 patch 从不同视角看同一 3D 点，由于光照、遮挡、透视和 context 的差异，它们的 DINO 特征本就不该完全相同。要求它们完全相同 = 在教模型忽略视角差异 = 丢弃位姿估计所需的信号。

**第二层：语义特征 ≠ 几何特征**
DINOv3 的语义特征让"门把手 ≈ 门把手"（跨实例泛化），匹配需要的几何特征要求"这个特定的门把手角落 ≠ 那个特定的门把手角落"（实例内区分）。对比学习试图把前者改造成后者 → 结果不是改造，是撕裂。

**第三层：训练目标与评估指标之间的三重不可微鸿沟**
训练优化余弦相似度 → MNN（不可微）→ USAC（不可微）→ 位姿分解（不可微）→ 角度误差。模型在训练时完全不知道"提高相似度"最终如何影响位姿。Precision 提升但 AUC 下降的分化现象是这一鸿沟的直接证据。

**第四层（最深）：预训练知识 vs 微调数据的毁灭性不对称**
DINOv3: 1.42 亿张图片。微调: 5596 对（0.0004%）。每一步梯度下降都在用极少的信息覆盖海量的预训练知识。不是微调，是微破坏。

### 核心教训

> 预训练视觉基础模型的强大源于其在海量数据上学习到的通用语义表示。任何试图用微量数据"修正"这个表示的操作，本质上都是在用极少的信息去覆盖极多的知识。正确的做法不是修改特征，而是在特征之上构建新的能力。
